## 一、现状拆解（你项目里的关键替换点）

1) 聊天请求入口（必须兼容）

- 文件：`app/src/main/java/ai/inmo/openclaw/data/repository/ChatService.kt`

- 当前固定请求：`POST /v1/chat/completions`

- 关键字段：

  - 请求：`model`, `stream=true`, `messages[{role,content}]`
  - 鉴权：`Authorization: Bearer <gateway token>`（可选）
  - 响应：SSE `data: {...}`，并按 `choices[0].delta.content` 增量解析
  - 结束：`data: [DONE]`

> 这意味着 Bootdo 或 Bootdo后面的转发层，必须输出与这个结构一致，否则前端会“看起来没响应”。

2) Provider 定义（inmoclaw来源）

- 文件：`domain/model/AiProvider.kt`

- `INMOCLAW` 当前：

  - `id = inmoclaw`
  - `baseUrl = ${BotNetworkModule.baseUrl}/im/ai/v4`
  - `defaultModels = [glm-4.7]`
  - `apiType = openai-completions`

3) 远端 bot 服务（非聊天主链）

- 文件：`BotApi.kt`
- 当前仅看到 `/im/bot/token-usage`，是配套统计，不是流式聊天主通道。

## 三、Bootdo 侧详细设计（你要做的核心）

1) 接口设计：先做“兼容层”

先实现一个兼容端点（名字可内部定义）：

- `POST /v1/chat/completions`（或 `/openai/v1/chat/completions` 再由网关转发）
- 必须支持 `stream=true` 的 SSE

请求兼容（最小必需）

```json
{
  "model": "glm-4.7",
  "stream": true,
  "messages": [
    {"role":"system","content":"..."},
    {"role":"user","content":"..."}
  ]
}
```

流式响应兼容（重点）

每行：

```javascript
data: {"choices":[{"delta":{"content":"你"}}]}

data: {"choices":[{"delta":{"content":"好"}}]}

data: [DONE]
```

这正是你 App 当前 `parseSse()` 的解析格式。

2) Bootdo 服务分层

建议在 Bootdo 中拆成：

- `AiController`：HTTP入参与SSE输出
- `AiChatService`：参数校验、上下文裁剪、业务策略
- `GlmClient`：实际调用智谱 GLM API
- `AiAuditService`：请求日志、耗时、token统计
- `AiRateLimitService`：Redis限流

3) GLM 4.7 调用策略

- 后端使用服务端 API Key（绝不下发到 App）
- 统一加 timeout、retry（有限重试）
- 错误映射：GLM错误 -> 统一业务错误码
- 输出增量 token 时，按 SSE 推给前端

## 五、你最容易踩坑的 8 个点（提前规避）

1. __SSE格式不一致__：必须 `data: ...\n\n`，且 `[DONE]` 完整。
2. __字段名不一致__：前端读的是 `choices[0].delta.content`。
3. __chunk 粒度太大__：用户感觉“卡顿”，建议小块推送。
4. __鉴权头冲突__：前端是 gateway token，不是 GLM key。GLM key仅Bootdo持有。
5. __超时配置不当__：流式 read timeout 需长于普通接口。
6. __模型名管理混乱__：`glm-4.7` 统一在后端映射，前端尽量不感知。
7. __审计缺失__：线上问题无法追踪，必须落日志（脱敏）。
8. __无回滚开关__：必须保留 old/new provider 切换开关。

agentclaw执行顺序：
1）在聊天框发送“天气如何”
App 侧先生成一条 `role=user` 的消息，创建 runId，进入发送流程。

2）消息先进入本机 Gateway（127.0.0.1:18789）**  
App 不是直连 8066，而是先把这次对话交给 OpenClaw Gateway（本地 18789）。

3）Gateway 组织完整 LLM 请求**  
Gateway 把 system prompt + 历史 + 你的 user 文本合并成 OpenAI 兼容 body，并发到 `http://192.168.110.37:8066/v1/chat/completions`。  
你日志里对应：`agentclaw REQ_8066_CHAT_*`，且 `stream` 被改成 `false`。

4）8066 接口接收并转发到 clawbootdo 上游**  
8066 只是兼容层/转发层，继续把请求转给 clawbootdo（再到实际模型）。

5）clawbootdo/模型生成结果并回传**  
正常：模型结果 → clawbootdo → 8066 → 18789。  
异常：这段没在超时窗口内返回，就会触发 `LLM request timed out.`。

6）18789 把结果（或错误）回给 App UI**  
- 成功：assistant 内容显示出来。  
- 失败：你看到 `state=error / aborted`，并显示超时。

链路是 App → 18789 → 8066 → clawbootdo/模型 → 8066 → 18789 → App
