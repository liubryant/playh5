# clawbootdo

## 项目说明

`clawbootdo` 是 OpenClaw 体系中的后端服务节点之一，主要承担 AI 请求的兼容转发与上游模型调用衔接能力。本文档重点说明 **agentclaw 一次典型问答请求的完整执行顺序**，用于排查“超时 / aborted / error”类问题以及理解端到端链路。

---

## agentclaw 执行顺序（完整链路）

以下以用户在聊天框输入：`天气如何` 为例。

### 1）App 侧创建用户消息并发起发送

- 用户在聊天框发送“天气如何”。
- App 先在本地会话里生成一条 `role=user` 的消息。
- 同时创建本次对话请求的 `runId`。
- 之后进入统一发送流程。

> 这一步只发生在 App 内部，尚未访问远端 8066。

---

### 2）消息先进入本机 Gateway（127.0.0.1:18789）

- App **不是**直接请求 `8066`。
- App 会先把这次对话交给本机 OpenClaw Gateway：`127.0.0.1:18789`。

> 也就是说，Gateway 是 App 与后端兼容层之间的第一跳。

---

### 3）Gateway 组织完整 LLM 请求

Gateway 在本机完成以下动作：

- 合并 `system prompt` + 历史对话 + 当前 user 文本（“天气如何”）。
- 按 OpenAI 兼容格式组装 request body。
- 发往：

```text
http://192.168.110.37:8066/v1/chat/completions
```

日志侧通常可看到类似标识：

- `agentclaw REQ_8066_CHAT_*`
- 且 `stream` 会被改为 `false`（由 Gateway 侧策略处理）

---

### 4）8066 接口接收并转发到 clawbootdo 上游

- `8066` 主要扮演兼容层 / 转发层。
- 它接收 Gateway 发来的 OpenAI 兼容请求。
- 然后继续把请求转给 `clawbootdo`（再由 clawbootdo 路由到实际模型服务）。

> 因此 8066 不是最终模型本体，而是请求中转与协议兼容节点。

---

### 5）clawbootdo / 模型生成结果并回传

#### 正常路径

模型生成结果后，响应会沿原路返回：

`模型结果 → clawbootdo → 8066 → 18789`

#### 异常路径

如果在超时窗口内没有返回成功响应，则触发：

- `LLM request timed out.`

这通常意味着：

- 上游模型处理超时，或
- clawbootdo 到模型的某段链路出现延迟 / 阻塞，或
- 8066 到 Gateway 回传未在期限内完成。

---

### 6）18789 将结果（或错误）回给 App UI

- **成功**：assistant 内容显示在聊天 UI。
- **失败**：可看到 `state=error / aborted`，并在界面提示超时。

---

## 总链路（必须牢记）

```text
App → 18789 → 8066 → clawbootdo/模型 → 8066 → 18789 → App
```

该链路是排查问题时的标准顺序，请按跳点逐段定位。

---

## 常见排查建议

1. **先看 App 与 18789**：确认请求是否创建 runId 并成功发出。
2. **再看 Gateway 到 8066**：确认 `REQ_8066_CHAT_*` 是否存在、参数是否完整、`stream=false` 是否符合预期。
3. **检查 8066 转发日志**：确认是否成功转发至 clawbootdo。
4. **检查 clawbootdo 与模型上游耗时**：重点关注是否出现超时、重试或网络抖动。
5. **回看 18789 最终状态**：成功应有 assistant 内容；失败通常为 `error/aborted` + timeout。

---

## 术语对照

- **App**：用户交互界面，负责构造本地消息、展示结果。
- **18789（Gateway）**：本机网关，负责拼装 LLM 请求与向 8066 发起调用。
- **8066**：OpenAI 兼容层 / 转发层。
- **clawbootdo**：后端服务逻辑层，负责与实际模型上游衔接。
- **模型上游**：最终执行推理并返回结果的模型服务。
